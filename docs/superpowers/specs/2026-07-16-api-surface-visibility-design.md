# Verifiable API surface ‚Äî OpenAPI page + constants coverage

**Date:** 2026-07-16
**Status:** Design ‚Äî awaiting review
**Author:** ≈Ýimon + Claude
**Related:** `2026-07-16-backend-frontend-compute-split-design.md` (parent design),
`2026-07-16-backend-split-f0-f1-cooking-pilot.md` (F1 pilot plan)

## 1. Problem

The backend split moves game data and computation into `core/`, served by
`api/compute?section=`. Two things are invisible today:

1. **The API contract.** There is no published description of `/api/compute`. An external
   consumer (the stated reason the compute layer is server-side at all ‚Äî parent design ¬ß3)
   has to read `api/compute.mjs` to learn the parameters.
2. **The data tables.** `core/data/` is becoming the canonical home of the game tables, but
   nothing renders them. There is no way to check that a table arrived intact and complete,
   or to see how much of the migration is actually done.

Both are the same defect the parent design attacks: truth that exists but cannot be checked
drifts. A hand-written API doc or a hand-listed constants page would become a *fourth* copy
of the truth ‚Äî the disease, not the cure.

## 2. Goals / Non-goals

**Goals**
- Publish the real `/api/compute` contract as OpenAPI, rendered as browsable docs.
- Render the canonical `core/data/` tables in the web UI so they can be eyeballed.
- Show honest migration coverage: which game tables are canonical in `core/data/` and which
  are still inline in `flowers.html`.
- **Drift must be mechanically impossible**: a test fails when code and description disagree.

**Non-goals**
- Documenting the 12 legacy endpoints (`proxy`, `farm-history`, `marketplace-*`, ‚Ä¶). They
  are internal to the web app. Only `/api/compute` is a public contract. (Decided.)
- Editing constants from the UI, or storing constants in Postgres. **Explicitly rejected** ‚Äî
  see ¬ß6.
- Moving all 160 data tables now. Coverage fills in as F2+ migrates each domain ‚Äî see ¬ß5.

## 3. Constraints (binding)

- **Vercel Hobby = 12 functions; `api/` is at 13 today.** Only files directly in `api/` count;
  `core/` and static files do not. **This feature must add ZERO functions.** Both new surfaces
  ride existing routes: the OpenAPI doc via `?section=openapi`, the constants via
  `?section=constants`, and the pages are static files.
- **`flowers.html`'s main script (line 2800) is a plain `<script>`, not a module.** It cannot
  `import` from `core/*.mjs`. Any constant the page needs must either stay inline or be
  fetched from the API. This is the reason coverage is incremental (¬ß5).
- `api/compute.mjs:8` rejects requests without `?farm=`. `openapi` and `constants` need no
  farm, so their branches must run **before** that check.
- No new npm dependencies. The repo has exactly one (`pg`).

## 4. Architecture

Both surfaces use one pattern: **single source in `core/` ‚Üí served via an existing route ‚Üí
rendered by a static page ‚Üí pinned by a drift test.**

```
core/api-spec.mjs        OpenAPI 3.1 document (JS object) ‚Äî the contract, single source
core/data/*.mjs          the canonical game tables (cooking, fishing, crafting, ‚Ä¶)
core/sections/constants.mjs   composer: enumerates core/data + the coverage inventory

api/compute.mjs          + section=openapi    ‚Üí core/api-spec.mjs        (no farm needed)
                         + section=constants  ‚Üí buildConstantsSection()  (no farm needed)
                         (both branch BEFORE the `farm required` check)

swagger.html             static page; Swagger UI ‚Üí /api/compute?section=openapi
flowers.html             CONSTANTS tab ‚Üí fetch /api/compute?section=constants
                         + a footer/MORE link to /swagger.html
```

Swagger UI assets load from unpkg CDN, matching the existing practice at `flowers.html:2639`
(lightweight-charts). No CSP is configured, so the CDN loads. If unpkg is down, only the docs
page fails to render ‚Äî no app or API impact.

## 5. Constants coverage ‚Äî the honest part

`section=constants` returns two things:

```
{
  tables:   { [name]: <the table, verbatim from core/data> },
  coverage: [ { name, lines, status: "core" | "inline", file } ]
}
```

`tables` is generated by enumerating `core/data/*.mjs` exports ‚Äî **not** a hand-written list.
A new table added to `core/data/` appears in the tab with no UI change.

`coverage` lists **all 160 game tables** known to exist (3614 lines total, inventoried from
`flowers.html`), each marked:
- `core` ‚Äî canonical in `core/data/`, rendered above. Today ~11 (cooking + fishing/crafting).
- `inline` ‚Äî still inline in `flowers.html`, not yet migrated.

This is why "everything in constants" is a *result*, not a task: a table can only become
canonical when its consumer migrates (¬ß3 ‚Äî the page cannot import `core/`). Each F2+ domain
flips its tables from `inline` to `core` and they appear automatically. The coverage view
makes the remaining work visible instead of pretending completeness.

Largest still-inline tables, for reference: `ITEM_IMAGE_MAP` (1220 lines), `BOOST_PARSE_RULES`
(347), `SKILL_TREE_DATA` (168), `BUMPKIN_GIFTS_DATA` (126), `WEARABLE_IDS` (78),
`SKILL_UPGRADES` (68), `BASE_STOCK` (18).

## 6. Rejected: constants in Postgres

Considered and rejected (user-approved rejection). Postgres exists here (`api/_db.js`, `pg`
pool) and serves tracking/history. Storing the game tables in it would:
- re-create two sources of truth (DB rows + the engine's expectations in code) ‚Äî the exact
  disease the parent design cures;
- force a database into the offline `node --test` gate, destroying the "verified to the cent
  without network" property the pilot depends on;
- add latency and a new failure mode to `/api/compute`, currently a pure function of
  (farm, settings);
- take versioning, review, and rollback away from git and hand them to `UPDATE`.

These are static game rules that ship with the engine. They belong in code.
Deploy-free constant *editing* is a real want, but it is a separate design, after the pilot.

## 7. Drift tests ‚Äî the acceptance gate

**`tests/core/api-spec.test.mjs`** parses the real `api/compute.mjs` and asserts:
1. every `section === "X"` literal in the handler appears in the spec's `section` enum, and
   vice versa;
2. every `req.query.X` read by the handler is a declared parameter in the spec, and vice versa.

Add a section or a query param without documenting it ‚Üí **the test fails**. This is the whole
point: the test must fail if the defect is reintroduced, not merely pass today.

**`tests/core/constants.test.mjs`** asserts:
1. every export of `core/data/*.mjs` appears in `tables` (no table silently missing from the tab);
2. the coverage inventory matches a fresh scan of `flowers.html` ‚Äî a table added inline, or one
   migrated to `core/`, without updating the inventory ‚Üí fails;
3. spot-check: a known table (e.g. `COOKING_RECIPES_DATA`) round-trips through the endpoint with
   its full key count (84 recipes).

Validation of the OpenAPI document itself is structural and hand-rolled (no npm validator ‚Äî
¬ß3, no new dependencies).

## 8. Sequencing

After the F1 pilot, so this does not collide with Tasks 11/12 which are actively rewriting
`flowers.html`:

- **Task 13 ‚Äî constants**: `core/sections/constants.mjs`, `section=constants`, CONSTANTS tab,
  `tests/core/constants.test.mjs`.
- **Task 14 ‚Äî swagger**: `core/api-spec.mjs`, `section=openapi`, `swagger.html`, footer link,
  `tests/core/api-spec.test.mjs`.

Task 14 is last because `section=constants` should already exist when the spec's section enum
is pinned, avoiding an immediate rewrite of the spec.

## 9. Risks

- **Coverage inventory maintenance.** The 160-table list must track `flowers.html`. Mitigated
  by test ¬ß7.2 scanning the file rather than trusting a checked-in list.
- **`ITEM_IMAGE_MAP` is 1220 lines** and would dominate the constants payload once migrated.
  When that domain lands, the section may need per-table lazy loading. Not a problem today.
- **CDN dependency for Swagger UI.** Accepted; precedent at `flowers.html:2639`.

## 10. Open questions

None blocking. Deferred: deploy-free constant editing (¬ß6); whether `section=all` should
include `constants` (probably not ‚Äî it is a debug surface, not farm data).
